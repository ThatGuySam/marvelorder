# Cloudflare Workers Migration Plan

Research checked on March 12, 2026.

## Original Prompt

> Research and Make a plan to migrate this app to Cloudflare Workers

## Goal

Move the public MarvelOrder site from Netlify to Cloudflare Workers with no regression in public page delivery, legacy URL handling, image rendering, or the current Bun-based build and test workflow.

## Non-Goals

- Reworking the content model, markdown structure, or TMDB/Fandom data pull logic unless migration work requires a small compatibility change.
- Adding Cloudflare storage products by default. KV, R2, D1, Durable Objects, or Images bindings should only be introduced where they clearly reduce risk or operational cost.
- Rebuilding the Tina authoring flow unless `/admin` is explicitly brought back into scope. The checked-in `/admin` artifact is not production-ready today.
- Converting the public site to full request-time SSR unless a later phase introduces a real feature need for it.

## Repo Findings

- The app is currently an Astro 6 project on Bun:
  - `astro: 6.0.3`
  - `packageManager: bun@1.2.15`
  - `@astrojs/netlify: 7.0.1`
- Current baseline on March 12, 2026:
  - `bun run build` passes
  - `bun run type-check` passes with hints only
  - `bun run test --run` passes with 43 tests
- The public site is already mostly static:
  - `src/pages/en` contains 434 markdown-backed content pages
  - `src/pages/stories` contains 34 story pages
  - `dist/` contains 491 files and is about 28 MB
- Netlify coupling is concentrated in a small number of places:
  - `astro.config.ts` uses `@astrojs/netlify`
  - `netlify.toml` defines publish settings, function packaging, and manual redirects
  - `src/helpers/url.ts`, `src/helpers/types.ts`, `src/config.ts`, and many markdown files hard-code `/.netlify/functions/...`
  - `src/pages/[slug]/functions/*` are compatibility shims that re-export the Astro API routes
- The main runtime-specific code is the image pipeline:
  - `src/pages/api/tmdb-image/[...slug].ts`
  - `src/pages/api/fanart/[...slug].ts`
  - both rely on `sharp`, `node-fetch`, `Buffer`, `image-size`, and `etag`
- `src/pages/404.astro` is dynamic today. It fetches `sitemap-0.xml` at request time and tries to redirect an old slug to a new canonical slug by matching the trailing numeric listing ID.
- Build-time helpers still use Node-oriented file access (`fs-extra`, `gray-matter`, `fast-glob`) in `src/helpers/node/*`, but those paths currently run during build, not per request, for most pages.
- `public/admin/index.html` is a Tina localhost shell that points to `http://localhost:4001`. If deployed as-is, `/admin` is not a usable production surface.

## External Research

| Research item | Source-backed finding | Repo implication |
| --- | --- | --- |
| Astro on Cloudflare | Astro’s Cloudflare adapter is for on-demand routes; if Astro is only being used as a static site builder, no adapter is required. Cloudflare’s Astro guide also documents a pure static deployment path with only `assets.directory = "./dist"`. | This repo should not start with full Astro SSR on Workers unless needed. The current app is already static-heavy. |
| Static assets plus Worker code | Workers Static Assets deploys static files and Worker code together. If an asset path matches a file, Cloudflare serves it without invoking Worker code; if no asset matches, the Worker handles the request and can still call `env.ASSETS.fetch(request)`. | This is a strong fit for the app: keep the site static, and use a thin Worker only for image endpoints and legacy URL handling. |
| `_redirects` and `_headers` | Cloudflare supports `_redirects` and `_headers` in the static asset directory, but those rules are not applied to responses generated by Worker code. | Manual Netlify redirects can move to `_redirects`, but legacy paths handled by the Worker must be redirected in code, not only in `_redirects`. |
| Workers limits | Current limits are 20,000 static files on free and 100,000 on paid, 25 MiB per static file, 3 MB gzip Worker size on free and 10 MB on paid, and 128 MB memory per isolate. | The current build output is comfortably within static asset limits. The real size risk is the runtime bundle if image logic pulls in large dependencies. |
| Node.js compatibility | Cloudflare supports a subset of Node APIs behind `nodejs_compat`, with unsupported APIs falling back to polyfills that may throw at runtime. `Buffer`, `fs`, and `path` are supported, but compatibility is not blanket parity with Node. | Turning on `nodejs_compat` may help narrow gaps, but it should not be treated as proof that native-heavy dependencies such as the current image stack will work at the edge. |
| Astro Cloudflare build/runtime behavior | Astro 6 on Cloudflare runs prerendered pages in `workerd` by default, but `prerenderEnvironment: 'node'` exists specifically for prerendered pages that depend on Node APIs or NPM packages incompatible with `workerd`. Astro also warns that some dependencies may need pre-compilation for the Cloudflare runtime. | If the repo later adopts `@astrojs/cloudflare`, it should start with `prerenderEnvironment: 'node'` because many prerendered pages import Node-oriented helpers. |
| Cloudflare image transforms | Cloudflare Images/Image Resizing supports width, quality, trim, border trimming, and `segment=foreground`, with built-in caching. | Cloudflare can likely replace part of the current image pipeline, but the repo’s exact `sharp` behavior is not a guaranteed drop-in. This needs a dedicated spike against representative URLs. |
| Netlify-to-Workers migration guidance | Cloudflare’s Netlify migration guide says Netlify-specific features are not supported directly and recommends using framework-specific guides when available. | The migration should replace Netlify-specific behavior instead of trying to emulate Netlify wholesale. |

## Recommendation

Use a static-first Workers architecture, not a full Astro SSR migration.

The current site already builds cleanly as a mostly prerendered Astro app, and almost all of the provider-specific runtime risk is concentrated in two image endpoints plus the legacy 404 redirect behavior. Cloudflare Workers Static Assets is a better first landing zone than `@astrojs/cloudflare` server mode because it preserves the current build shape, keeps content rendering build-time only, and minimizes `workerd` compatibility risk.

Recommended target shape:

1. Keep Astro output primarily static and deploy `dist/` through Workers Static Assets.
2. Add a thin custom Worker for:
   - `/api/tmdb-image/*`
   - `/api/fanart/*`
   - legacy `/.netlify/functions/*` compatibility routes if needed
   - unmatched legacy slug redirects based on listing ID
3. Decide image handling in an early spike:
   - preferred: pre-generate or store transformed images outside request-time runtime work
   - fallback: implement a Cloudflare Images-backed approximation and accept only after visual validation
4. Revisit full Astro-on-Workers SSR only if the app later needs sessions, actions, or other true request-time features.

## Rollout Plan

### Stage 0: Confirm scope and freeze acceptance samples

- Decide whether `/admin` is in scope for this migration.
- If `/admin` is out of scope, explicitly exclude it from launch acceptance and consider redirecting or disabling it during cutover.
- Capture representative acceptance samples before any migration work:
  - home page
  - one `/en/...` page with a TMDB logo
  - one `/en/...` page with a fanart logo
  - one `/stories/...` page
  - one missing legacy slug that currently resolves through `404.astro`
  - 15-20 image URLs covering:
    - `transparent=0`
    - default transparent mode
    - `crop.top`
    - `crop.bottom`
    - TMDB and fanart sources
- Record current response behavior, cache headers, and screenshot baselines for those pages and images.

### Stage 1: Remove Netlify-only public URL assumptions

- Introduce provider-neutral image route helpers, for example:
  - `/api/tmdb-image/...`
  - `/api/fanart/...`
- Replace Netlify-specific path generation in:
  - `src/helpers/url.ts`
  - `src/helpers/types.ts`
  - `src/config.ts`
  - any helper or component code that still emits `/.netlify/functions/...`
- Codemod markdown and story frontmatter so new content output stops depending on `/.netlify/functions/...`.
- Keep backward compatibility for old URLs separately in the Worker layer instead of baking provider-specific paths into content forever.
- Delete the duplicate `src/pages/[slug]/functions/*` shims once the new route strategy is in place and validated.

### Stage 2: Replace request-time sitemap parsing with a build-time redirect manifest

- Stop fetching `sitemap-0.xml` on every 404.
- Add a small build-time generator that creates a listing-ID redirect manifest from current page filenames, for example:
  - `{ "198178": "/en/wonder-man-198178/", ... }`
- Update the eventual Worker to:
  - inspect the trailing numeric ID in unmatched `/en/...` requests
  - redirect to the canonical current slug when the ID exists in the manifest
  - otherwise serve the static 404 page
- Keep explicit one-off marketing or renamed-page redirects in `_redirects` where static rules are enough.

### Stage 3: Decide the image runtime strategy

- Run a dedicated spike against the current image API contract.
- Preferred direction: stop doing `sharp` work at request time.
  - Evaluate generating transformed assets during the existing build or data-pull flow and storing them under `public/` or an external object store such as R2.
  - If asset generation becomes deterministic, serve those files directly through Static Assets and remove the runtime image endpoints entirely.
- Fallback direction: use Cloudflare image transforms behind a Worker route.
  - Map supported options such as width, quality, format, trim, and caching.
  - Test whether `trim=border` and `segment=foreground` are good enough replacements for the current transparent-logo behavior.
  - Define a clear acceptance threshold before switching production traffic.
- Do not assume `nodejs_compat` makes the current `sharp`-based implementation safe to run on Workers.

### Stage 4: Add Cloudflare deployment scaffolding

- Add `wrangler.jsonc` with:
  - project name
  - current `compatibility_date`
  - `assets.directory = "./dist"`
  - `assets.binding = "ASSETS"` if a custom Worker is used
  - `observability.enabled = true`
- If the Worker code still needs supported Node APIs, add `compatibility_flags = ["nodejs_compat"]`.
- Move static redirects from `netlify.toml` into `public/_redirects`.
- Add `public/_headers` only for rules that apply to static asset responses.
- Add deployment scripts, for example:
  - `deploy`
  - `preview:workers`
  - `wrangler:types` if Worker bindings are typed

### Stage 5: Implement the thin Worker

- Add `src/worker.ts` and keep the routing narrow:
  - handle legacy `/.netlify/functions/tmdb-image/*`
  - handle legacy `/.netlify/functions/fanart/*`
  - handle new `/api/tmdb-image/*`
  - handle new `/api/fanart/*`
  - handle unmatched legacy content slugs through the build-time redirect manifest
  - delegate any matching static asset request to `env.ASSETS.fetch(request)`
  - return the static 404 page when no redirect or dynamic route matches
- Keep manual redirect logic in Worker code for any routes that must execute before or instead of static asset routing.
- If image generation remains runtime-based, add cache-aware responses and explicit error behavior for unsupported transforms.

### Stage 6: CI/CD and secret mapping

- Keep the existing GitHub Actions data-refresh workflow unless there is a separate reason to move it.
- Decide whether deploys will run through:
  - GitHub Actions plus Wrangler
  - Workers Builds
- Prefer GitHub Actions at first if you want tighter control over Bun and Deno versions.
- Split secrets by lifecycle:
  - build-time only secrets stay in GitHub Actions or the chosen build environment
  - Worker runtime secrets move to Cloudflare secrets
- Audit each current environment variable and classify it:
  - build-time scripts: `TMDB_API_KEY`, `DISNEY_API_PREFIX`, `AWS_*`
  - Tina: `NEXT_PUBLIC_TINA_CLIENT_ID`, `TINA_TOKEN`
  - runtime Worker: only add what the Worker actually needs after the image strategy is chosen

### Stage 7: Cutover and cleanup

- Deploy a preview Worker on a non-production domain.
- Compare preview against production on the acceptance set from Stage 0.
- Attach the production custom domain only after image parity and legacy redirect behavior are acceptable.
- After stable production traffic:
  - remove `@astrojs/netlify`
  - remove `netlify.toml`
  - remove `dev-netlify`
  - remove any remaining Netlify path comments and types
  - remove temporary legacy compatibility routes only after checking logs and analytics

## Validation Gates

- Must pass: `bun run test --run`
- Must pass: `bun run type-check`
- Must pass: `bun run build`
- Must pass: preview deploy or dry-run deploy through Wrangler
- Must verify on a Cloudflare preview:
  - `/`
  - `/marvel-cinematic-universe-in-timeline-order/`
  - one `/en/...` page using a TMDB-derived image
  - one `/en/...` page using a fanart-derived image
  - one `/stories/...` page
  - one old slug that should redirect by numeric ID
  - one current manual redirect from `netlify.toml`
- Must compare representative image outputs before cutover:
  - rendered dimensions
  - transparency behavior
  - crop behavior
  - cache headers
- Must confirm the deployment stays within Cloudflare limits:
  - static asset count
  - static asset file size
  - Worker bundle size

## Deliverables

- A Cloudflare deployment path for the public site using Workers Static Assets
- `wrangler.jsonc` and deploy scripts
- A thin Worker for dynamic compatibility needs
- Provider-neutral image URLs in helpers and content generation
- A build-time legacy redirect manifest replacing runtime sitemap parsing
- A documented decision on image handling:
  - pre-generated assets
  - or Cloudflare image transforms with accepted tradeoffs
- Removal plan for Netlify-specific config and routing once traffic is stable

## Risks And Open Questions

- The image endpoints are the main migration risk. The current `sharp` behavior may not have exact parity with Cloudflare’s transform APIs.
- `/admin` is not deployable as-is from this repo because the checked-in file points to `localhost:4001`. Decide whether to fix it, move it elsewhere, or keep it out of scope.
- If old `/.netlify/functions/...` image URLs have meaningful external traffic, compatibility routes may need to stay longer than expected.
- If the team wants full Astro-on-Workers SSR later, the repo’s Node-oriented build helpers mean the first adapter pass should use `prerenderEnvironment: 'node'` and test `workerd` compatibility deliberately.
- Cloudflare free-tier Worker size and CPU limits may be too tight if runtime image processing remains heavy. Pre-generation or a paid plan may be necessary.
- The current `netlify.toml` redirects are few, but the listing-ID redirect behavior is dynamic. That should stay in code or a generated manifest, not be expanded into a brittle manual redirect file.
